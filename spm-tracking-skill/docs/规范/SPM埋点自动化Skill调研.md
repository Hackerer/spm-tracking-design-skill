## 三、SPM埋点自动化 Skill 设计方案（V2.0版）

基于您作为资深数据产品专家的专业反馈，优化后的设计方案如下：

### 3.1 核心设计原则

| 原则 | 说明 |
|------|------|
| **需求深度理解** | 充分理解需求文档中每一个细节，不遗漏任何业务逻辑 |
| **设计稿结构化** | 理解Figma截图之间的页面关系，提取每个页面的交互元素 |
| **业务参数优先** | 只关注业务参数，SDK级别参数（uid/timestamp/device等）由底层自动处理 |
| **全局一致性** | SPM设计和参数设计需遵循全局规范，确保跨页面/模块的一致性 |
| **Human in the Loop** | 任何不确定的环节必须主动确认，不自行猜测 |
| **文档沉淀机制** | 每个需求完成后，询问是否沉淀为全局规范文档 |

---

### 3.2 优化后的 Workflow 设计

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    SPM埋点自动化设计 Workflow (V2.0)                     │
└─────────────────────────────────────────────────────────────────────────┘

[输入]
├── 需求文档 (PRD/需求说明) - 包含完整业务逻辑
└── Figma 设计稿截图文件 - 多页面关系图 + 页面细节图

═══════════════════════════════════════════════════════════════════════════
                           执行流程 + Human Check
═══════════════════════════════════════════════════════════════════════════

▼ Step 1: 需求深度解析 ⚠️ HUMAN CHECK
│  ├─ 理解业务流程：用户从进入页面到完成目标的完整路径
│  ├─ 识别业务实体：商品、订单、用户等核心对象
│  ├─ 提取关键指标：转化率、点击率、停留时长等
│  └─ 确认点：[请确认需求理解是否完整？是否有遗漏的业务场景？]
│
▼ Step 2: Figma 设计稿结构化理解 ⚠️ HUMAN CHECK
│  ├─ 理解页面关系：首页→列表页→详情页→下单页的跳转关系
│  ├─ 提取页面元素：每个页面的核心交互元素（按钮/列表/卡片/浮层等）
│  ├─ 识别可埋点位置：所有用户可点击/可交互的区域
│  └─ 确认点：[请确认页面结构和元素理解是否正确？]
│
▼ Step 3: 公共业务参数确认
│  ├─ 已有业务公共参数：用户提供（如 product_id, order_id 等）
│  ├─ 页面级公共参数：当前页面的通用业务参数
│  └─ 注：不关注SDK级参数（uid/timestamp/session等）
│
▼ Step 4: SPM 位置模型设计 ⚠️ HUMAN CHECK
│  ├─ a级(业务线): 全局统一规范（如 shop/cart/user/pay）
│  ├─ b级(页面/模块): 根据设计稿页面结构确定
│  ├─ c级(位置区域): 根据元素所在区域确定（如 banner/list/recommend）
│  └─ d级(具体元素): 根据交互元素确定（如 buy_btn/add_cart）
│  └─ 确认点：[请确认SPM命名是否符合全局规范？是否有冲突？]
│
▼ Step 5: 参数设计 ⚠️ HUMAN CHECK
│  ├─ 命名一致性：相同含义的参数使用统一命名
│  ├─ 数据类型：string/number/boolean/array
│  ├─ 枚举值定义：如 status: [pending/success/fail]
│  └─ 确认点：[请确认参数设计是否符合已有规范？]
│
▼ Step 6: 埋点文档生成
│  ├─ 埋点清单表格：事件ID + 事件名 + SPM + 参数 + 触发时机
│  ├─ 页面SPM映射表：页面与a.b级的对应关系
│  ├─ 参数定义表：参数名 + 类型 + 必填/选填 + 说明 + 枚举值
│  └─ 代码示例：埋点调用示例
│
▼ Step 7: 文档沉淀询问 ⚠️ HUMAN CHECK
│  └─ 确认点：[是否需要沉淀以下文档？]
│      ├── 埋点设计规范（新增事件/参数模板）
│      ├── SPM全集文档（新增的SPM枚举值）
│      └── 参数管理文档（新增/变更的参数定义）
│
[输出]
└── 埋点方案文档.md
```

### 3.3 输入输出定义

#### 输入

| 输入项 | 说明 | 要求 |
|--------|------|------|

## 6. 代码示例

### 6.1 立即购买点击
```javascript
// SPM: shop.product_detail.banner.buy_btn
SPM.track('buy_btn_click', {
  spm: 'shop.product_detail.banner.buy_btn',
  product_id: '123456',
  sku_id: 'sku_001',
  price: 9900
});
```

### 6.2 加入购物车

  price: 9900,
  quantity: 1
});
```

## 四、Human in the Loop 机制设计

### 4.1 强制确认节点

| 节点 | 确认内容 | 跳过条件 |
|------|---------|---------|
| Step 1 完成后 | 需求理解是否完整准确 | 用户确认可直接跳过 |
| Step 2 完成后 | 页面结构和元素理解是否正确 | 用户确认可直接跳过 |
| Step 4 完成后 | SPM命名是否符合全局规范 | 已有全局规范文档 |
| Step 5 完成后 | 参数设计是否一致 | 已有参数规范文档 |

### 4.2 确认话术示例

```
⚠️ [请确认] 需求理解检查
我理解的需求要点如下：
1. 核心业务流程：用户浏览商品 → 加入购物车 → 结算 → 支付
2. 需要埋点的页面：商品详情页、购物车页、结算页、支付结果页
3. 关键交互点：立即购买、加购、选择规格、提交订单

请确认以上理解是否正确？如有遗漏或偏差，请指出。
```
